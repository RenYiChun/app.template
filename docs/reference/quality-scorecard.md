# App Template 框架质量评分卡

本文档记录收缩 Dataforge 之后的仓库状态，评分基准为 10 分制，重点关注当前保留的安全基座、Flow 引擎与微服务能力。

---

## 一、综合评分卡

| 维度 | 评分 | 说明 |
|------|:---:|------|
| **模块化与职责分离** | 8/10 | 主线更聚焦，但仍存在历史包袱与文档残留风险 |
| **异常处理体系** | 8/10 | 统一异常契约仍然清晰 |
| **缓存策略** | 8/10 | Caffeine 基线明确，容量与 TTL 可控 |
| **可观测性（指标）** | 8/10 | Flow 与安全链路指标覆盖较完整 |
| **可观测性（日志）** | 7/10 | MDC 追踪可用，但缺少分布式 tracing 体系 |
| **安全体系** | 7/10 | JWT/Opaque Token 双模可用，但边界策略仍需加强 |
| **Flow 引擎设计** | 9/10 | 背压、公平调度、虚拟线程、可插拔存储仍是最大亮点 |
| **线程与资源管理** | 8/10 | 线程命名和双重关闭保障较完善 |
| **配置治理** | 8/10 | 类型安全、启动校验和摘要日志仍然有效 |
| **韧性能力** | 5/10 | 仅有可选重试，无熔断、限流、降级 |
| **测试质量** | 6/10 | Flow 测试较强，其他模块仍偏弱 |
| **构建工程化** | 6/10 | 有静态检查，但缺少 CI/CD，JaCoCo 默认跳过 |
| **文档覆盖** | 7/10 | 主文档齐全，但仓库收缩后仍需持续清理历史叙事 |

**综合评分：7.2 / 10**

---

## 二、按优先级列出的不足

### 高优先级

| # | 不足项 | 位置/说明 |
|---|--------|-----------|
| 1 | **无 CI/CD** | 质量保障仍依赖本地执行 |
| 2 | **CSRF 策略较粗** | 浏览器场景与纯 API 场景没有充分分层 |
| 3 | **无限流机制** | API 层缺少统一限流保护 |

### 中优先级

| # | 不足项 | 位置/说明 |
|---|--------|-----------|
| 4 | **JaCoCo 默认跳过** | 覆盖率门槛未默认纳入日常构建 |
| 5 | **后端无 i18n** | 错误消息仍以硬编码中文为主 |
| 6 | **无 API 版本控制** | 破坏性变更缺少标准演进策略 |
| 7 | **README 仍偏模板化叙事** | 需要进一步向 Flow 主线收敛 |

### 低优先级

| # | 不足项 |
|---|--------|
| 8 | 工具类 Javadoc 仍不均衡 |
| 9 | 无熔断器与降级策略 |
| 10 | `template-api`、`template-cloud`、`template-oauth2-service` 仍需更完整模块 README |

---

## 三、做得好的方面

- 模块边界经过收缩后更清晰
- Flow 引擎的运行时设计仍具备差异化
- 统一配置属性与启动校验保留了较高工程价值
- Micrometer 指标与健康检查对线上排障仍有帮助
- 异常处理契约统一，降低了业务层样板代码

---

## 四、后续优化建议顺序

1. 引入 CI/CD，并把核心模块编译与单测纳入默认流水线
2. 启用 JaCoCo 检查，至少先覆盖 `template-flow`
3. 为安全配置补充分场景的 CSRF 与限流策略
4. 继续清理历史叙事，把仓库定位收敛到 Flow 与基础设施
5. 为保留模块补充更完整的独立 README 与最小接入示例
