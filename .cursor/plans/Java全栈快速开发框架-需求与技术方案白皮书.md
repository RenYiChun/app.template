# Java 全栈快速开发框架 · 完整需求与技术方案白皮书

---

## 第一部分：战略定位

### 1.1 核心愿景

**「声明即应用」**——让 Java 开发者仅通过编写带注解的 Java 类，在编译期自动生成完整可运行的前后端应用，获得手写代码般的掌控力与低代码般的效率。

### 1.2 本质定义

本框架不是运行时框架，而是**标准化 CRUD 的编译器**：

- **输入**：带注解的 Java 领域对象 + 页面描述类
- **处理**：编译期 APT + 模板渲染
- **输出**：可读、可调试、可纳入 Git 管理的标准 Spring Boot + Vue 3 代码

### 1.3 竞争护城河

| 维度         | 传统低代码 | 手写开发 | 本框架   |
| ------------ | ---------- | -------- | -------- |
| 交付效率     | ⭐⭐⭐⭐⭐   | ⭐⭐      | ⭐⭐⭐⭐   |
| 代码可控     | ⭐         | ⭐⭐⭐⭐⭐  | ⭐⭐⭐⭐⭐  |
| 二次开发体验 | ⭐⭐        | ⭐⭐⭐⭐   | ⭐⭐⭐⭐   |
| 长期维护成本 | ⭐⭐⭐       | ⭐⭐⭐     | ⭐⭐⭐⭐   |
| 技术债务     | 高         | 中       | 低       |

**差异化核心**：不锁死、不黑盒、不留技术债；默认采用「生成与手写分离」的扩展策略，保证用户代码永不被误覆盖。

## 第二部分：功能性需求

### 2.1 元模型体系

```java
// 领域模型驱动
@Domain(table = "sys_user")
public class User {
    @Id @GeneratedValue
    private Long id;

    @Column(length = 50, nullable = false)
    @FormField(label = "用户名", required = true)
    private String username;

    @Column(nullable = false)
    @FormField(label = "密码", type = "password")
    private String password;

    @FormField(label = "邮箱", type = "email")
    private String email;
}

// 页面即对象
@Page(title = "登录页", layout = "centered")
public class LoginPage {
    @FormField(label = "账号", required = true)
    private String username;

    @FormField(label = "密码", type = "password", required = true)
    private String password;

    @FormField(label = "记住我", type = "checkbox")
    private Boolean rememberMe;
}
```

## 第三部分：推荐架构与扩展策略（最佳实践）

### 3.1 默认策略：生成与手写分离（策略 C）

**本框架将「生成与手写分离 + 接口与钩子扩展」作为推荐最佳实践。**

- **生成代码**：输出到独立目录或独立模块（如 `generated/` 或 `*-generated` 模块），用户**不直接编辑**；需要升级时由生成器整体覆盖。
- **用户代码**：位于 `src/` 或独立应用模块，仅**依赖**生成层；扩展通过实现接口、挂载钩子、编写 BFF 或适配层完成。
- **效果**：生成器永远不改写用户文件，无「合并错误」风险；行为可预测、文档简单、信任成本最低。

### 3.2 推荐扩展方式

| 方式               | 说明                                                                 | 适用场景                         |
| ------------------ | -------------------------------------------------------------------- | -------------------------------- |
| **接口 + 默认实现** | 生成 `UserService` 接口 + `UserServiceImpl`；Controller 依赖接口，用户可提供自己的 Impl 或子类 | 业务逻辑扩展、替换实现           |
| **生命周期钩子**   | 生成代码调用 `BeforeSaveHandler`、`AfterLoadHandler` 等；用户实现并注册为 Spring Bean | 审批、审计、计算字段等           |
| **生成到独立模块** | 生成产物在单独模块，应用模块只依赖该模块，不修改生成文件             | 希望「生成层可随时全量覆盖」的项目 |
| **BFF / 适配层**   | 前端不直连生成 API，而是访问用户自己的 BFF；BFF 内部再调生成后端或手写服务 | 强定制前端、聚合接口、多后端     |
| **模板覆盖 / 继承** | 项目内提供自定义 Freemarker 模板覆盖默认模板（v2.0）                  | 统一团队风格、定制生成风格       |
| **配置驱动**       | `.fastgen/config.yml` 控制生成哪些层、UI 库、是否生成测试等           | 多项目、多技术栈复用同一套元模型 |
| **注解透传**       | 生成器不识别的自定义注解原样进入元数据，模板中按需使用                 | 列显示、校验、权限等细粒度扩展   |

### 3.3 可选能力：手写保护区与智能合并

面向「希望生成代码与手写代码共处同一文件」的高级用户，提供**可选**能力；不作为默认路径。

- **手写保护区**
  - 约定：每文件**至多一对** `USER_CUSTOM_START` / `USER_CUSTOM_END`；禁止嵌套与交叉。
  - 行为：解析异常时**不修改该文件**并告警；提供 `fastgen:validate` 校验标记配对与嵌套。

- **变更预览**
  - `mvn fastgen:diff` 输出结构化变更说明与可选文本 diff。
  - 支持非交互模式（如 `-Dfastgen.apply=true` 或 `--yes`）及明确退出码：0=无变更/已应用，1=有未应用变更，2=错误，便于 CI 集成。

- **智能字段合并**（v1.5，可选）
  - 仅在「结构化元数据 + 约定范围内的列表/表单」上做「追加列、追加表单项」等有限合并；
  - 超出约定范围（如用户随意改 .vue）则视为手写，不再对该文件做智能合并；
  - 具体语义在 v1.5 设计阶段用 2～3 个场景的规则与示例写清，避免二义。

## 第四部分：非功能性需求

### 4.1 核心质量属性

| 属性     | 指标                     | 实现策略                                           |
| -------- | ------------------------ | -------------------------------------------------- |
| 性能     | 10 个实体 &lt; 3 秒（目标） | APT 并行、模板缓存；MVP 优先正确性                 |
| 可读性   | 生成代码 ≈ 手写代码      | 缩进、注释、空行；可配置 code style（如 EditorConfig/Checkstyle） |
| 可调试   | 源码级调试               | 输出到 `generated-sources` 或约定目录              |
| 可维护   | 模板与逻辑分离           | Freemarker + 模板继承；APT 只产元数据，模板只做渲染 |
| 兼容性   | 向前兼容                 | 元数据版本与快照 schema 约定                       |

### 4.2 安全约束

- 默认生成登录拦截与安全配置扩展点（可与既有安全库集成）。
- 密码字段自动使用 password 类型，不落库明文。
- 不生成敏感配置（如数据库密码），仅生成配置模板或占位说明。
- API 路径遵循 REST 与安全规范。

### 4.3 明确排除范围

- 不替代复杂业务逻辑（仅做标准化 CRUD）。
- 不承诺可视化拖拽设计。
- 不绑定特定云厂商。
- 不承诺运行时动态表单。
- 不解决分布式事务等高级场景。

## 第五部分：技术方案要点

### 5.1 APT 与模板的职责划分

- **APT 阶段**：仅负责从注解中提取并校验元数据，产出**结构化元数据**（JSON 或内存模型）；不做业务逻辑、不做字符串拼接。
- **模板阶段**：仅根据元数据做**渲染**；不在模板中写复杂 Java 逻辑，保持模板可读、可替换。
- **元数据 schema**：在 v1.0 即约定 `.fastgen/snapshot-*.json` 等快照的字段定义，便于 v1.5 智能合并与迁移脚本演进。

### 5.2 手写保护区规范（可选能力）

- 每文件**至多一对** `USER_CUSTOM_START` / `USER_CUSTOM_END`。
- 禁止嵌套、禁止交叉；多对或异常时**跳过该文件合并并告警**。
- 提供 `fastgen:validate` 校验标记，建议在 CI 中执行。

### 5.3 变更预览与 CI

- `fastgen:diff` 支持非交互模式与退出码约定（见 3.3）。
- 生成代码目录与命名规范在 MVP 即固定，便于后续多模块与 v1.5 合并语义。

### 5.4 多模块与目录约定

- MVP 明确「单应用 + 单模块」边界；目录与包结构预留多模块扩展（如 `generated-sources` 归属、子模块生成触发方式）。
- 多模块下 `.fastgen` 的放置策略（每模块一个 vs 根目录一个）在 v1.0 约定，v1.5 实现。

### 5.5 与既有模板库集成（可选）

- 生成项目可**可选依赖**现有安全/基础设施库（如 template-api），生成「带登录、RBAC、审计」的 CRUD 而非裸 Controller。
- 生成代码预留 Security 配置扩展点、RBAC 注解等，便于与既有 OAuth2/RBAC 体系对接。

## 第六部分：开发者体验设计

### 6.1 脚手架

目标：三分钟从零到可运行。

```bash
# 方案 A：Maven Archetype
mvn archetype:generate -DarchetypeGroupId=io.fastgen \
  -DarchetypeArtifactId=fastgen-archetype \
  -DgroupId=com.company -DartifactId=my-project

# 方案 B：CLI
npx create-fastgen-app my-app
cd my-app

# 一键启动
mvn compile
mvn spring-boot:run    # 后端（另一终端）
npm run dev            # 前端

# 浏览器打开 http://localhost:5173，登录页与用户管理已就绪
```

脚手架内置：预置 `User.java`、`LoginPage.java`；内嵌 H2（零配置）；前后端联调代理；简短 README 与 3 条命令说明。

### 6.2 生成策略（与推荐策略一致）

| 文件/目录类型           | 处理策略                                   |
| ----------------------- | ------------------------------------------ |
| 生成层（独立模块/目录） | 每次生成可覆盖；用户不直接编辑             |
| 用户层（实现类、钩子、BFF） | 生成器永不覆盖                         |
| 可选：同文件手写保护区 | 仅当启用且校验通过时合并；否则跳过并告警   |

### 6.3 工作流适配

- IDE：将生成目录标记为源码根。
- Git：生成文件头部注明来源（如 `Generated from User.java`）。
- Code Review：以元数据与用户代码为主，生成代码默认可信。

## 第七部分：生产级能力

### 7.1 多环境

- 仅生成配置**模板**（如 `application-dev.yml.example`），不生成含敏感信息的配置；用户自行复制并填写，生成器不覆盖已存在配置。

### 7.2 数据库迁移

- 生成 Flyway 风格迁移脚本（如 `V20240213__add_avatar_to_user.sql`），不直接执行 ALTER。

### 7.3 测试

- 自动生成单元测试基架（如 Controller 空壳测试）；用户可在独立测试类或保护区中补充业务断言（v1.5 细化）。

## 第八部分：演进路线图

### MVP v1.0 — 验证核心价值

- 目标：少量 Java 类 → 一个完整可运行应用。
- 内容：`@Domain`、`@Page` 基础注解；APT 元数据提取；生成 User CRUD 后端（接口 + 默认实现）+ Vue 列表/表单；登录页；Maven Archetype 脚手架；**推荐策略 C（生成与手写分离 + 接口/钩子）**；元数据 schema 与目录规范约定。
- 可选：手写保护区（单文件一对 + validate）+ 变更预览（含非交互与退出码）。
- 成功标准：新项目 3 分钟内可运行，敢用、能用。

### v1.5 — 信任与可选增强

- 目标：从「能用」到「敢长期用」。
- 内容：元数据快照与快照管理；单元测试生成增强；Flyway 迁移脚本；Swagger 文档；多模块支持；**可选**智能字段合并（限定语义与场景）。

### v2.0 — 生态扩展

- 目标：可配置、可扩展、可升级。
- 内容：多 UI 模板（Element Plus / AntDV）；模板继承与自定义模板；存量项目逆向（DB → 注解类）；框架升级助手；IDE/VS Code 插件。

## 总结

本框架定位为 **Spring Boot + Vue 生态中的「声明即应用」编译器**：不替代手写，而是把重复劳动自动化；不与低代码拼功能广度，而在「可控效率」上做到极致。

**护城河**建立在可预测、可接管的信任体系上：

- **推荐最佳实践**：生成与手写分离 + 接口与钩子扩展（策略 C），用户代码永不被误覆盖。
- **可选能力**：手写保护区、变更预览、有限智能合并，满足高级用户需求并明确约束与边界。
- 脚手架保证首次即可成功；透明生成与规范保证随时可接管；演进路线保证长期可维护。
